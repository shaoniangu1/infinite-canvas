# 新模型 / 新渠道接入指引

本文档用于约束 AI 编码助手新增模型或渠道时的设计和实施方式。目标是保持模型参数、用户交互、请求协议和任务生命周期职责清晰，避免把正式能力降级为散落的字符串脚本或组件分支。

开始任务前必须先阅读本文档，并以官方 API 文档为唯一字段依据。

## 1. 核心分层

```text
Model Profile
  -> 模型目录、能力类型、参数 UI、素材要求、必填与枚举校验

Provider Adapter
  -> 鉴权、上传、请求体转换、异步任务、错误和结果归一化

Model Call Script
  -> 单模型请求覆盖、临时验证和私有中转站兼容
```

运行时优先级必须保持：

```text
模型调用脚本 > Provider Adapter > OpenAI / Gemini 默认实现
```

调用脚本只覆盖请求执行，不负责声明参数 UI、素材角色或产品校验。

## 2. 接入前必须取得的信息

开始编码前必须取得官方 API 文档，并整理成字段表。至少包含：

- 请求地址、HTTP 方法和鉴权方式。
- 准确模型 ID，不使用未经文档确认的别名。
- 请求字段、类型、含义、必填或可选状态、默认值和枚举。
- 提示词是否必填，空提示词是否允许。
- 图片、视频、音频输入的角色、数量、格式、大小和时长限制。
- 本地素材是否需要先上传，以及上传接口和返回 URL 结构。
- 同步返回或异步任务流程，包括任务 ID、状态和轮询间隔。
- 成功、失败、取消、超时和结果响应示例。
- 分辨率、比例、时长、生成数量等模型差异。

无法从官方文档确认的字段不得自行猜测。第三方示例只能作为补充证据，不能覆盖官方文档。

## 3. 先判断接入类型

| 请求协议 | 参数和素材要求 | 接入方式 |
| --- | --- | --- |
| OpenAI / Gemini 兼容 | 与现有 UI 完全一致 | 仅配置渠道和模型，不改代码 |
| 非标准协议 | 现有 UI 足够 | 先用调用脚本验证；长期使用时升级为 Provider Adapter |
| 标准协议 | 有专属参数或素材要求 | 新增或扩展 Model Profile，并补请求映射 |
| 非标准协议 | 有专属参数或素材要求 | Provider Adapter + Model Profile |

不要因为服务商名称不同就新增 `ApiCallFormat`。只要协议真实兼容，应继续使用 `openai` 或 `gemini`。

### 3.1 只新增普通模型

如果模型与现有协议、参数和素材要求完全一致：

1. 在配置页拉取或手动增加模型。
2. 指定 `image`、`video`、`text` 或 `audio` 能力。
3. 验证对应工作台和生成配置节点能选中该模型。

这种情况不需要 Provider Adapter，也不需要模型专属参数声明。

### 3.2 已有渠道新增特殊模型

如果请求渠道不变，但模型存在专属字段、选项或素材要求：

1. 将内置模型加入 `web/src/services/ai/model-profiles.ts`。
2. 扩展对应能力的 Model Profile。
3. 让设置面板根据 Profile 渲染，不在组件中直接堆模型 ID 判断。
4. 在 UI 层和请求构造层复用同一套校验语义。
5. 为官方请求示例增加纯请求体断言。

当前视频模型入口是 `web/src/services/ai/video-model-profiles.ts`。图像、文本或音频出现真正的模型专属参数时，应建立同职责的声明层，不要把新分支直接写入页面组件。

### 3.3 新增非标准渠道

只有满足以下任一条件时，才新增正式 Provider Adapter：

- 使用非 OpenAI / Gemini 鉴权或接口路径。
- 需要服务商专属文件上传。
- 需要异步任务提交、轮询和结果解析。
- 错误结构、模型目录或请求体长期不同。
- 该渠道属于项目正式支持范围，需要用户零脚本配置使用。

首次探索可以使用调用脚本验证协议。验证通过且确定长期支持后，再迁移为类型化、可测试的 Provider Adapter；不要把验证脚本直接当作正式实现。

## 4. Model Profile 职责

Model Profile 负责定义产品能力，而不是发送网络请求。至少需要表达：

- 模型 ID、显示名称、Provider 和能力类型。
- 任务类型，例如文生视频、图生视频、参考视频或 motion-control。
- 图片、视频和音频的最小/最大数量及角色。
- 当前模型需要显示的参数字段。
- 字段控件类型、显示名称、枚举和必填状态。
- 提示词、素材组合和字段取值校验。

强制要求：

- 与模型无关的参数不能显示。
- 必填项必须在 UI 标识，并在提交前拦截。
- 提交层必须再次校验，不能只依赖 UI。
- 未选择的可选字段不发送，不提供无意义的“使用默认”选项。
- 模型匹配优先使用准确 ID 或受控模型族规则，不使用范围过大的关键字猜测。

Kling motion-control 和 KIE Seedance 2 可作为现有实现参考：

- `web/src/services/ai/video-model-profiles.ts`
- `web/src/services/ai/seedance-model-settings.ts`
- `web/src/components/video-settings-panel.tsx`

## 5. Provider Adapter 职责

Provider Adapter 放在 `web/src/services/ai/providers/`，只处理服务商协议差异：

- 鉴权头和 Provider URL。
- 参考素材读取和上传。
- 规范化参数到服务商请求体的映射。
- 异步任务提交、轮询、取消和超时。
- Provider 错误信息提取。
- 结果 URL、Blob 或 base64 的归一化。

共用异步任务生命周期优先复用 `web/src/services/ai/media-task-runtime.ts`。纯请求体组装应独立为无副作用函数，例如 `web/src/services/ai/providers/kie-video-payload.ts`，便于直接对照官方示例测试。

不要在 `web/src/services/api/image.ts`、`video.ts`、`audio.ts` 或 React 组件中复制上传、轮询和错误解析逻辑。API 文件只保留最小分发入口。

新增正式 Provider 通常需要检查：

1. `web/src/stores/use-config-store.ts`：仅在确有新协议时扩展 `ApiCallFormat`、默认地址和持久化规范化。
2. `web/src/components/layout/channel-editor-drawer.tsx`：增加渠道协议选项。
3. `web/src/components/layout/app-config-modal.tsx`：增加协议显示名称。
4. `web/src/services/ai/model-profiles.ts`：增加内置模型目录。
5. `web/src/services/ai/providers/<provider>-provider.ts`：实现正式 Adapter。
6. `web/src/services/api/`：在调用脚本之后增加最小 Provider 分发。

## 6. 调用脚本使用边界

调用脚本适用于：

- 新接口快速验证。
- 用户自己的私有网关。
- 使用频率低且现有参数 UI 足够的模型。
- 标准协议只有少量请求字段差异的中转站。
- Provider API 临时变化时的短期覆盖。

调用脚本不适用于：

- 需要专属参数 UI 的长期核心模型。
- 需要复杂文件上传和素材规则的正式渠道。
- 需要稳定异步任务、统一错误和任务 ID 的渠道。
- 需要项目内置模型目录和零配置使用的渠道。

脚本通过 `new Function` 在浏览器中执行，并能取得当前渠道 API Key。只能使用可信脚本，不得在文档、日志或错误信息中输出密钥和完整鉴权头。

## 7. 标准实施顺序

1. 保存官方文档地址、请求示例和字段表。
2. 按第 3 节判断接入类型，说明为什么需要或不需要 Adapter/Profile。
3. 添加模型目录和能力分类。
4. 添加或复用 Model Profile。
5. 先编写请求体和校验的定向测试。
6. 编写纯请求体构造函数。
7. 必要时实现 Provider Adapter，并复用上传和任务运行时。
8. 在 `services/api/` 中增加最小分发，保持脚本覆盖优先。
9. 验证工作台、生成配置节点、失败提示和重试使用同一行为。
10. 更新 `CHANGELOG.md`、`pending-test.mdx`，并检查 `todo.mdx` 是否需要调整。

## 8. 请求与交互强制规则

- 空数组、空字符串和空 webhook 不得随意发送。
- 可选字段只有用户选择或文档要求时才发送。
- 本地参数必须转换为 API 文档规定的真实枚举值。
- 素材上传失败时不得继续创建付费生成任务。
- 提交前错误必须使用用户可理解的中文提示。
- Provider 返回的 `msg`、`message`、`error` 和任务失败原因应优先展示。
- 轮询必须支持取消和明确超时，超时信息尽量包含任务 ID。
- 结果解析只兼容官方或已取得真实样本的结构，不无依据堆叠猜测分支。
- 项目尚未上线，不增加旧配置迁移和兼容代码，除非用户明确要求。
- 未经用户明确授权，不调用真实付费生成接口验证。

## 9. 最低测试范围

新增模型或渠道至少覆盖与改动相关的以下项目：

- 模型进入正确 Provider 和能力列表。
- 当前模型只显示相关参数字段。
- 必填、可选、枚举、提示词和素材数量校验。
- 官方请求示例能够得到结构一致的请求体。
- 未选择的可选字段和空素材数组不会发送。
- 分辨率、比例、时长和数量边界值正确。
- 上传失败不会创建生成任务。
- 成功、失败、取消、超时和真实结果结构可以解析。
- 调用脚本存在时优先于 Provider Adapter。
- 不配置脚本时仍走正式 Adapter 或默认协议。
- 生成失败后的重试保留模型参数和参考素材。

现有定向测试参考：

- `web/src/services/ai/video-model-profiles.test.mjs`
- `web/src/services/ai/provider-adapters.test.mjs`
- `web/src/services/api/model-plugin.test.mjs`

按项目规则运行与改动匹配的定向测试；不主动执行全量构建，除非用户明确要求。

## 10. 禁止做法

- 不因服务商名称不同就复制一个 Provider Adapter。
- 不把 Provider 请求体拼装放进 React 组件。
- 不在多个 API 文件复制相同上传和轮询代码。
- 不用调用脚本替代正式核心渠道的 Profile 和 Adapter。
- 不让组件和请求层各自维护互相不一致的必填规则。
- 不显示模型不支持的参数，再依赖 Provider 忽略。
- 不为单个模型建立庞大抽象，也不为未来假设提前增加字段。
- 不输出 API Key、Token、完整鉴权头或浏览器本地凭证。

## 11. 完成汇报

AI 完成接入后必须说明：

- 使用的官方文档地址。
- 模型字段表及必填/可选结论。
- 选择的接入类型及理由。
- 修改的 Model Profile、Provider Adapter 和 API 分发入口。
- 实际请求体与官方示例的对应关系。
- 定向测试命令和结果。
- 是否调用了真实接口；未调用时明确说明剩余风险。
- 尚待人工验证的 UI、上传、轮询和付费生成行为。
